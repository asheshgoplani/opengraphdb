<objective>
Add all-or-nothing bulk import mode to OpenGraphDB that rolls back the entire import on any record failure.

Purpose: IMEX-01 requires that import supports an all-or-nothing bulk mode. Currently, imports batch records into multiple transactions (default 10,000 per batch). A single corrupt record either aborts the current batch (default) or skips it (`--continue-on-error`). Neither mode guarantees the database is completely unchanged on failure. This plan adds `--atomic` mode that wraps the entire import in a single transaction.

Output: `--atomic` flag on `ImportCommand` and `ImportRdfCommand`, modified `ImportBatcher` to support single-transaction mode, rollback semantics on any failure.
</objective>

**Step 4: Modify ImportBatcher::flush to abort on any error in atomic mode (line ~4224)**

In the flush method, when `atomic_mode` is true and any record application fails, roll back the entire transaction and return an error that indicates the import was rolled back:

Find the error handling section inside flush. The current code looks like:

```rust
Err(err) => {
    if self.continue_on_error {
        skipped_records = skipped_records.saturating_add(1);
```

In atomic mode, `continue_on_error` is always false (enforced by `conflicts_with`), so the existing abort path works correctly. When `flush()` returns an `Err`, the `WriteTransaction` is dropped without being committed, triggering the automatic rollback via the `Drop` impl. No special handling is needed here.

However, for clarity, we can add a more descriptive error message when in atomic mode. Modify the error handling:

```rust
Err(err) => {
    if self.continue_on_error {
        skipped_records = skipped_records.saturating_add(1);
        continue;
    }
    // In atomic mode, the WriteTransaction will be dropped
    // without commit, triggering automatic rollback
    if self.atomic_mode {
        return Err(CliError::Runtime(format!(
            "atomic import rolled back: record {} failed: {err}",
            self.progress.processed_records + applied_records + skipped_records + 1
        )));
    }
    return Err(CliError::Runtime(format!("failed to import record: {err}")));
```

**Step 6: Verify atomic rollback semantics**

The key insight is that the existing `WriteTransaction` Drop impl already handles rollback. In atomic mode:
1. All records accumulate in `self.pending` (push never flushes mid-stream)
2. `finish()` calls `flush()` once with all records
3. `flush()` creates one `WriteTransaction` via `self.db.begin_write()`
4. If any record fails, `flush()` returns `Err`, causing the transaction to be dropped without `commit()`
5. The `Drop` impl calls `rollback_in_place()`, undoing all changes

This means the database is left completely unchanged on failure, which is exactly the all-or-nothing semantic.

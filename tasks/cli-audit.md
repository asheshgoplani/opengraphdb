# OpenGraphDB CLI Audit Report

**Date:** 2026-02-24
**Audited against:** [clig.dev](https://clig.dev/) guidelines (condensed in `create-cli` skill)
**CLI binary:** `ogdb-cli` (crate: `crates/ogdb-cli`)
**Source:** `crates/ogdb-cli/src/lib.rs` (~14,000 lines) + `main.rs` (14 lines)
**Version:** 0.1.0

---

## Summary

The CLI has a solid functional foundation: correct exit code conventions (0/1/2), proper stdout/stderr separation, a `--format` global flag for machine-readable output (json/jsonl/csv/tsv), TTY detection in the shell, and a working REPL with history and tab completion. However, it falls significantly short of clig.dev standards in **discoverability** and **self-documentation**. Every subcommand and nearly every argument lacks a description, making the help output almost useless for new users. There is no `--verbose`/`--quiet`, no `--dry-run`, no color support or `NO_COLOR` awareness, no shell completion generation, no environment variable support, and the database path argument is inconsistent across commands (some require it positionally, some allow `--db`, some do both via a complex injection hack). The CLI is functional for the author but hostile to anyone discovering it for the first time.

**Overall grade: 2.1/5** (functional but not user-ready)

## Issues Found

### CRITICAL

#### C1. Zero subcommand descriptions in main help listing

- **Severity:** CRITICAL
- **Category:** Help
- **Current behavior:** Every subcommand appears with a blank description in `--help`:
  ```
  Commands:
    init
    info
    query
    shell
    import
    export
    import-rdf
    export-rdf
    backup
    checkpoint
    schema
    stats
    metrics
    mcp
    serve
    create-node
    add-edge
    neighbors
    incoming
    hop
    hop-in
    help         Print this message or the help of the given subcommand(s)
  ```
  Only `help` has a description (auto-generated by clap).
- **Guidelines say:** "Help text is part of UX. Lead with examples; show common flags/commands first."
- **Source:** `lib.rs:54-81`, the `Commands` enum has no `about` annotations on any variant.
- **Recommended fix:** Add `#[command(about = "...")]` to every variant in the `Commands` enum:
  ```rust
  #[command(about = "Initialize a new database")]
  Init(InitCommand),
  #[command(about = "Show database file metadata")]
  Info(DbPathCommand),
  #[command(about = "Execute a Cypher query")]
  Query(QueryCommand),
  // ... etc
  ```

#### C2. Zero argument/option descriptions on most commands

- **Severity:** CRITICAL
- **Category:** Help
- **Current behavior:** Arguments render as bare placeholders with no explanation:
  ```
  Arguments:
    <path>
    <src>
    <dst>

  Options:
    --commands <q1;q2;...>
    --script <path>
    --online
    --compact
    --bolt
    --http
    --grpc
    --stdio
    --schema-only
    --continue-on-error
  ```
  None of these have help text. The only documented options across the entire CLI are the two global flags (`--format` and `--db`).
- **Guidelines say:** Every flag and argument should be self-describing.
- **Source:** All command structs (`lib.rs:83-308`) have `#[arg()]` attributes but only `--format` and `--db` include `help = "..."`.
- **Recommended fix:** Add `help = "..."` to every `#[arg()]` attribute. Priority targets (most confusing without docs): `ServeCommand` (what do `--bolt`/`--http`/`--grpc` do?), `McpCommand` (what is `--stdio`?), `BackupCommand` (what is `--online`? `--compact`?), `ImportCommand` (`--continue-on-error`?), `ShellCommand` (`--commands`? `--script`?), `AddEdgeCommand` (`<src>` and `<dst>` are node IDs but this is never stated).

### HIGH

#### H1. Inconsistent database path argument pattern

- **Severity:** HIGH
- **Category:** Args/Flags
- **Current behavior:** Three different patterns exist:
  1. **Optional positional + `--db` fallback** (init, info, shell, schema, stats, metrics, checkpoint, mcp, serve, create-node): `[path]` with `required_unless_present = "db_path"`
  2. **Required positional** (query, import, export, import-rdf, export-rdf, add-edge, neighbors, incoming, hop, hop-in): `<path>` always required as first positional, plus a 50-line `try_parse_with_injected_db_path` hack (`lib.rs:485-496`) that re-parses the entire arg vector with the `--db` value injected as the first positional arg
  3. **Different arg names** (backup): uses `<src-path>` and `<dst-path>` positionally (no `--db` support at all)
- **Guidelines say:** "Be consistent across subcommands. Prefer flags over positional args for clarity and future flexibility."
- **Recommended fix:** Standardize all commands to accept `[path]` as optional positional with `--db` fallback. Remove the `try_parse_with_injected_db_path` injection hack. For commands needing additional paths (import, export, backup), use flags: `--src`, `--dst`, `--input`, `--output`.

#### H2. Missing `--quiet` / `--verbose` global flags

- **Severity:** HIGH
- **Category:** Args/Flags
- **Current behavior:** No way to suppress output or get more detail. Every command has fixed output verbosity. Import always prints its full summary. Serve always prints "listening on...".
- **Guidelines say:** Standard flags: `-q, --quiet` (less output), `-v, --verbose` (more output). "Don't leak developer-only output to normal users; reserve for verbose/debug."
- **Source:** Grepping for `verbose|quiet` in lib.rs returns zero hits.
- **Recommended fix:** Add global `-q, --quiet` and `-v, --verbose` flags to the `Cli` struct. Quiet: suppress success messages, only output essential data (e.g., node_id from create-node). Verbose: show timing info, database open details, query plans, page counts.

#### H3. Missing `--dry-run` for destructive operations

- **Severity:** HIGH
- **Category:** Robustness
- **Current behavior:** `init`, `import`, `import-rdf`, `create-node`, `add-edge`, `backup`, `checkpoint` all modify state immediately with no preview option.
- **Guidelines say:** "Provide safe dry run/preview; confirm scary actions." Standard flag: `-n, --dry-run` (preview only).
- **Source:** Grepping for `dry.run|force` in lib.rs returns zero hits.
- **Recommended fix:** Add `--dry-run` to `import` and `import-rdf` (show detected format, record count, estimated nodes/edges without writing). For `backup`, add `--dry-run` to show source size and destination path without copying.

#### H4. No environment variable support

- **Severity:** HIGH
- **Category:** Config
- **Current behavior:** No env vars are recognized at all. Grepping for `OPENGRAPHDB_|OGDB_` returns zero hits. No `NO_COLOR` check.
- **Guidelines say:** "Precedence: flags > env > project config > user config. Respect NO_COLOR, TERM=dumb."
- **Recommended fix:** Support at minimum:
  - `OGDB_DB` or `OPENGRAPHDB_DB`: default database path (can use clap's `env = "OGDB_DB"`)
  - `OGDB_FORMAT`: default output format
  - `NO_COLOR`: disable color (when color is added)
  - `OGDB_LOG_LEVEL` or `RUST_LOG`: verbosity control

#### H5. No shell completion generation

- **Severity:** HIGH
- **Category:** Help / Distribution
- **Current behavior:** No `completions` subcommand. No `--generate-completion`. The interactive REPL has rustyline completion for Cypher keywords, but the outer CLI has no shell tab completion.
- **Guidelines say:** "Shell completion story: install/discoverability; generation command or bundled scripts."
- **Recommended fix:** Add `clap_complete` crate. Add a `completions` subcommand:
  ```
  opengraphdb completions bash > /etc/bash_completion.d/opengraphdb
  opengraphdb completions zsh > ~/.zfunc/_opengraphdb
  opengraphdb completions fish > ~/.config/fish/completions/opengraphdb.fish
  ```

#### H6. Raw error messages lack context and actionable guidance

- **Severity:** HIGH
- **Category:** Errors
- **Current behavior:** Many errors surface raw OS or library messages:
  ```
  io error: No such file or directory (os error 2)
  ```
  No indication of which file, what operation failed, or what to do about it. Tested: `opengraphdb info /nonexistent.ogdb` and `opengraphdb init /nonexistent/path/test.ogdb` both produce this raw message.
- **Guideline says:** "Catch and rewrite expected errors for humans. For every error, tell the user what to do next."
- **Source:** The `From<DbError> for CliError` impl at `lib.rs:380-384` does a bare `.to_string()` on the database error with no wrapping. In contrast, `handle_import` (`lib.rs:4807-4810`) already has an excellent pattern: `"error: database not found at '{db_path}'. Run 'ogdb init <path>' first."`.
- **Recommended fix:** Wrap all `Database::open()` and `Database::init()` call sites with contextual messages like `handle_import` already does. Add the file path and a recovery hint to every IO error.

### MEDIUM

#### M1. No color support or `NO_COLOR` awareness

- **Severity:** MEDIUM
- **Category:** Output
- **Current behavior:** Zero color anywhere. No `--no-color` flag. `NO_COLOR` env var not checked. `TERM=dumb` not checked.
- **Guidelines say:** "Use color sparingly; disable when not a TTY, NO_COLOR set, TERM=dumb, or --no-color."
- **Recommended fix:** When adding color (error prefixes in red, REPL prompt, table headers), respect `NO_COLOR`, `TERM=dumb`, and add `--no-color` global flag. Even without color features, document the output is plain text.

#### M2. `serve` command does not display defaults in help

- **Severity:** MEDIUM
- **Category:** Help
- **Current behavior:** `serve --help` shows `--bind <ADDR>`, `--port <PORT>` with no indication of defaults. The default protocol (MCP) is not stated. Default ports vary by protocol:
  - Bolt/MCP: 7687
  - HTTP: 8080
  - gRPC: 7689
  - Default bind: 127.0.0.1 (HTTP/MCP), 0.0.0.0 (Bolt/gRPC)

  None of this is visible to the user.
- **Source:** `resolve_serve_bind_addr` at `lib.rs:2808-2826` contains all the defaults.
- **Guidelines say:** "Default should be right for most users."
- **Recommended fix:** Add help text showing defaults:
  ```
  --port <PORT>  Server port [default: 7687 for bolt/mcp, 8080 for http, 7689 for grpc]
  --bind <ADDR>  Bind address [default: 127.0.0.1 for http/mcp, 0.0.0.0 for bolt/grpc]
  ```
  Also document that MCP is the default protocol when no `--bolt`/`--http`/`--grpc` is specified.

#### M3. No examples in help text

- **Severity:** MEDIUM
- **Category:** Help
- **Current behavior:** Zero examples in any `--help` output across all 20 subcommands.
- **Guidelines say:** "Lead with examples. Show 5-10 example invocations covering common flows; include piped/stdin examples."
- **Recommended fix:** Add `#[command(after_long_help = "EXAMPLES:\n  ...")]` to the main command and key subcommands. At minimum:
  ```
  EXAMPLES:
    opengraphdb init ./mydb.ogdb
    opengraphdb --db ./mydb.ogdb query "MATCH (n) RETURN n LIMIT 10"
    opengraphdb import ./mydb.ogdb data.json
    opengraphdb shell ./mydb.ogdb
    opengraphdb serve --http ./mydb.ogdb
    cat queries.cypher | opengraphdb shell --db ./mydb.ogdb
  ```

#### M4. No progress indicators for long operations

- **Severity:** MEDIUM
- **Category:** Robustness
- **Current behavior:** Import/export of large files gives no progress feedback. The backup callbacks explicitly discard progress info: `db.backup_online(dst_path, |_copied, _total| {})` at `lib.rs:829`.
- **Guidelines say:** "Show progress for long tasks (interactive only); don't interleave logs confusingly."
- **Recommended fix:** When stderr is a TTY, print progress for import (record count every N records), export (node/edge progress), and backup (use the existing `|copied, total|` callback parameters). Print to stderr to avoid polluting stdout.

#### M5. `--format` as a global flag is confusing for RDF commands

- **Severity:** MEDIUM
- **Category:** Args/Flags
- **Current behavior:** There is a `normalize_rdf_format_alias` hack (`lib.rs:543-556`) that silently converts `--format` to `--rdf-format` when inside `import-rdf` or `export-rdf` commands. This means `--format` means "output format" globally but "RDF serialization format" in RDF commands. This silent rewrite breaks user expectations.
- **Guidelines say:** "Be consistent. Avoid ambiguous flag overloading."
- **Recommended fix:** Remove the alias hack. Keep `--format` strictly for output format and `--rdf-format` as the only way to specify RDF serialization. Document clearly in help text.

#### M6. `--format` flag shown on commands that ignore it

- **Severity:** MEDIUM
- **Category:** Args/Flags
- **Current behavior:** `--format` is a global flag, so it appears in help for every subcommand including `init`, `backup`, `checkpoint`, `mcp` where it has no effect. Running `opengraphdb --format json init ./mydb` silently ignores the flag.
- **Guidelines say:** "Validate early; fail fast with good error messages."
- **Recommended fix:** Either move `--format` to per-subcommand level where it applies, or warn when used with commands that ignore it.

#### M7. Subcommand naming inconsistency

- **Severity:** MEDIUM
- **Category:** Naming / Subcommands
- **Current behavior:** Mixed naming patterns across 20 commands:
  - **Verbs:** `init`, `query`, `serve`, `backup`, `import`, `export`
  - **Bare nouns:** `shell`, `schema`, `stats`, `metrics`, `neighbors`, `incoming`
  - **Verb-noun compound:** `create-node`, `add-edge`, `import-rdf`, `export-rdf`
  - **Abbreviated:** `hop`, `hop-in`
  - **Noun-ish:** `checkpoint`, `mcp`
- **Guidelines say:** "Consider noun-verb or verb-noun; pick one and stick to it. Be consistent across subcommands."
- **Recommended fix:** Pick a convention and apply it. Options:
  - **Nested nouns** (like `docker`): `node create`, `edge add`, `db info`, `db stats`
  - **Verb-first flat** (like `git`): keep `create-node`, `add-edge`, rename `neighbors` to `get-neighbors`, `incoming` to `get-incoming`, `schema` to `show-schema`, `stats` to `show-stats`

#### M8. No stdin support for `query` command

- **Severity:** MEDIUM
- **Category:** Args/Flags
- **Current behavior:** `query <path> [query]` only accepts the query string as a positional arg. No way to pipe a query:
  ```
  echo "MATCH (n) RETURN n" | opengraphdb query mydb.ogdb   # doesn't work
  ```
  Workaround: use `shell --commands` or pipe through `shell`.
- **Guidelines say:** "Support `-` for stdin/stdout when input/output is a file. Assume your output becomes someone else's input."
- **Recommended fix:** When `[query]` is omitted and stdin is not a TTY, read query from stdin. The `shell` command already does TTY detection (`lib.rs:1615`), so the pattern exists.

#### M9. Server "listening" messages go to stdout

- **Severity:** MEDIUM
- **Category:** Output
- **Current behavior:** Bolt server uses `println!("{listening}")` at `lib.rs:2840`. The listening message is operational status, not primary data output.
- **Guidelines say:** "Primary data to stdout; diagnostics/errors to stderr."
- **Recommended fix:** Change `println!` to `eprintln!` for all server startup messages (bolt, http, mcp).

#### M10. No "next command" suggestions after mutations

- **Severity:** MEDIUM
- **Category:** Help / Output
- **Current behavior:**
  - `init` prints: `initialized /path (format_version=1, page_size=4096)` with no suggestion.
  - `import` prints counters but no suggestion to verify.
  - `create-node` prints: `node_id=0` with no suggestion.
- **Guidelines say:** "Suggest next commands in workflowy tools."
- **Recommended fix:** After key operations, suggest logical next steps:
  - After `init`: `"Try: opengraphdb shell /path"`
  - After `import`: `"Verify: opengraphdb stats /path"`
  - After `create-node`: `"Add edges: opengraphdb add-edge --db /path <src> <dst> --type KNOWS"`

### LOW

#### L1. No link to documentation or issue tracker in help

- **Severity:** LOW
- **Category:** Help / Documentation
- **Current behavior:** Help output contains no URL, no repository link, no "report bugs at" message.
- **Guidelines say:** "Link to a support path (repo/issues/docs)."
- **Recommended fix:** Add to the main `#[command()]` attribute:
  ```rust
  #[command(after_help = "Docs: https://github.com/.../opengraphdb\nBugs: https://github.com/.../opengraphdb/issues")]
  ```

#### L2. Version output doesn't include build info

- **Severity:** LOW
- **Category:** Help
- **Current behavior:** `opengraphdb 0.1.0` and nothing else.
- **Recommended fix:** Consider adding git commit hash and build date: `opengraphdb 0.1.0 (abc1234 2026-02-24)`. Useful for bug reports and debugging.

#### L3. Binary name `opengraphdb` is long

- **Severity:** LOW
- **Category:** Naming
- **Current behavior:** Binary name is `opengraphdb` (11 chars). Every invocation requires typing this. The internal `APP_NAME` constant is `"opengraphdb"`.
- **Guidelines say:** "Keep it short but not cryptic; easy to type matters."
- **Recommended fix:** Ship an `ogdb` alias/symlink or rename the binary to `ogdb`. The Cargo binary name is already `ogdb-cli`.

#### L4. `--props` format is not documented

- **Severity:** LOW
- **Category:** Help
- **Current behavior:** `--props <k=type:value;...>` is the only hint. The actual format `name=string:Alice;age=i64:30` with supported types (bool, i64, f64, string, bytes, vector) is buried in `parse_property_value_literal` at `lib.rs:5850-5876`.
- **Recommended fix:** Add help text: `--props "key=type:value;..." where type is one of: string, i64, f64, bool, bytes, vector (e.g. "name=string:Alice;age=i64:30")`

#### L5. History file location is not XDG-compliant

- **Severity:** LOW
- **Category:** Config
- **Current behavior:** Shell history saved to `~/.ogdb_history` (`lib.rs:1508-1511`).
- **Guidelines say:** "Follow XDG base directories for user-level config when applicable."
- **Recommended fix:** Use `$XDG_DATA_HOME/opengraphdb/history` (default `~/.local/share/opengraphdb/history`), falling back to `~/.ogdb_history` for backwards compatibility.

#### L6. `backup` has no confirmation for overwriting destination

- **Severity:** LOW
- **Category:** Robustness
- **Current behavior:** `backup src.ogdb dst.ogdb` will overwrite `dst.ogdb` silently. No `--force` flag.
- **Guidelines say:** "Destructive operations: interactive confirmation + non-interactive requires --force."
- **Recommended fix:** Check if destination exists. If so, require `--force` to overwrite (or prompt when stdin is a TTY).

#### L7. `add-edge` requires positional `<path>` but `create-node` uses optional `[path]`

- **Severity:** LOW
- **Category:** Args/Flags
- **Current behavior:** `add-edge <path> <src> <dst>` requires the database path positionally. `create-node [path]` allows `--db` instead. This inconsistency exists because `add-edge` has multiple positional args and the injection hack handles it.
- **Recommended fix:** Make both use `--db` as the canonical path mechanism. Remove positional `<path>` from `add-edge`.

#### L8. No `--json` shorthand

- **Severity:** LOW
- **Category:** Args/Flags
- **Current behavior:** Must type `--format json`. No `--json` shorthand.
- **Guidelines say:** "Add `--json` for machine output."
- **Recommended fix:** Add `--json` as an alias for `--format json`. This is the most common machine-readable invocation pattern.

#### L9. Bolt/gRPC default bind to `0.0.0.0`

- **Severity:** LOW
- **Category:** Robustness
- **Current behavior:** `serve --bolt` defaults to `0.0.0.0:7687`, exposing to all network interfaces. Source: `lib.rs:2821-2824`.
- **Recommended fix:** Default to `127.0.0.1` for all protocols. Require explicit `--bind 0.0.0.0` to expose externally. Safer default.

#### L10. No man page generation

- **Severity:** LOW
- **Category:** Documentation
- **Current behavior:** No man page, no `--help` pager for long output.
- **Guidelines say:** "Provide terminal docs; consider man pages where sensible."
- **Recommended fix:** Add `clap_mangen` crate for man page generation. Low priority for 0.1.

## What's Already Good

- **Exit codes** follow convention (0 success, 1 runtime error, 2 usage error) with distinct `CliResult::ok`, `CliResult::user_error`, `CliResult::runtime_error`
- **stdout/stderr separation** is mostly correct (errors to stderr, data to stdout)
- **`--format`** with 5 output modes (table/json/jsonl/csv/tsv) is excellent for scriptability
- **`--db`** as global flag alternative to positional path is a thoughtful design
- **Interactive shell** with rustyline, persistent history, and Cypher keyword tab completion
- **TTY detection** for interactive vs. batch mode in shell command
- **Error message for missing path** is user-friendly: `"database path required: provide <path> or --db"`
- **Import error for missing database** has an actionable message: `"Run 'ogdb init <path>' first"`
- **`--continue-on-error`** on import commands is a good robustness feature
- **Input validation** on `--page-size` (power of two, >= 64) and `--batch-size` (>= 1) is solid
- **clap v4 with derive** provides a good foundation for all recommended improvements
- **MCP stdio mode** properly uses BufReader/BufWriter for protocol compliance

**Step 1: Create `skills/data-import/SKILL.md`**

Role definition: You are a data import expert for OpenGraphDB. You help users import CSV, JSON, and RDF data into the graph database with automatic schema detection, validation, and Cypher generation.

Content should cover:
- **Your approach**: Analyze the source data, detect schema, validate quality, generate import Cypher, execute import
- **Import workflow**:
  1. Examine the source file (read headers, sample rows)
  2. Detect format: CSV (headers, delimiter), JSON (array vs objects, nested structure), RDF (format variant)
  3. Infer graph schema: which columns/fields become nodes, edges, properties
  4. Check existing database schema via `browse_schema` to avoid conflicts
  5. Validate data quality: nulls, type consistency, uniqueness of IDs
  6. Generate import Cypher (or delegate to import_rdf for RDF)
  7. Execute import in batches using `execute_cypher`
  8. Verify import via `list_datasets` and sample queries
- **Format-specific handling**: Reference @rules/format-detection.md, @rules/import-patterns.md, @rules/validation-checks.md
- **Key principle**: Always MERGE on unique identifiers, not CREATE, to prevent duplicates on re-run. Make imports idempotent.

Target: ~80-100 lines.

**Step 2: Create `skills/data-import/rules/format-detection.md`**

Schema detection rules for each format:

**CSV Detection:**
1. Read the first line as headers
2. Read 5-10 sample rows to infer types:
   - All integers → Integer type
   - All numbers with decimals → Float type
   - "true"/"false" → Boolean type
   - ISO date strings → Date type
   - Everything else → String type
3. Look for ID columns: columns named `id`, `_id`, `uuid`, or containing `_id` suffix
4. Look for relationship indicators: columns named `*_id` (foreign keys), columns containing comma-separated values (multi-value)
5. Detect delimiter: comma (default), tab, semicolon, pipe
6. Handle encoding: assume UTF-8, warn if BOM detected

**JSON Detection:**
1. Top-level array: each element is a record (node candidate)
2. Top-level object with arrays: each key-array pair is a collection (label candidate)
3. Nested objects: potential embedded nodes or properties
4. Arrays within records: potential multi-value properties or relationship lists
5. Look for `type`, `label`, `category` fields → label candidates
6. Look for `*_id`, `source`, `target`, `from`, `to` fields → relationship candidates

**RDF Detection:**
1. File extension: .ttl (Turtle), .nt (N-Triples), .rdf/.xml (RDF/XML), .jsonld (JSON-LD)
2. Content sniffing: `@prefix` → Turtle, `<` and `>` with `.` → N-Triples, `<?xml` → RDF/XML
3. Delegate to `import_rdf` MCP tool for all RDF formats
4. After import, run `browse_schema` to report what was created

**Schema Inference Output Format:**
```
Detected Schema:
  Nodes:
    - :Person (name: String, age: Integer, email: String) [unique: name]
    - :Company (name: String, industry: String) [unique: name]
  Relationships:
    - (:Person)-[:WORKS_AT]->(:Company) via column 'company_name'
  Notes:
    - Column 'id' will be used as merge key
    - 3 rows have null 'email' values (will be imported as null)
```

Target: ~100-120 lines.

**Step 3: Create `skills/data-import/rules/import-patterns.md`**

Cypher generation patterns for importing data:

**CSV Import (single entity type):**
```cypher
// For each row in CSV:
MERGE (n:Person {name: $name})
ON CREATE SET n.age = $age, n.email = $email
ON MATCH SET n.age = $age, n.email = $email
```

When OpenGraphDB does not support Cypher parameters via HTTP, use string interpolation with proper escaping:
```cypher
MERGE (n:Person {name: 'Alice'})
ON CREATE SET n.age = 30, n.email = 'alice@example.com'
```

**CSV Import (with relationships):**
```cypher
// First pass: create nodes
MERGE (p:Person {name: 'Alice'})
ON CREATE SET p.age = 30;

// Second pass: create relationships
MATCH (p:Person {name: 'Alice'}), (c:Company {name: 'Acme'})
CREATE (p)-[:WORKS_AT]->(c)
```

**JSON Import (batch via UNWIND):**
```cypher
UNWIND [{name: 'Alice', age: 30}, {name: 'Bob', age: 25}] AS row
MERGE (n:Person {name: row.name})
ON CREATE SET n.age = row.age
```

**JSON Import (via POST /import API):**
For large datasets, use the POST /import endpoint with JSON records:
```json
[
  {"type": "node", "labels": ["Person"], "properties": {"name": "Alice", "age": 30}},
  {"type": "node", "labels": ["Person"], "properties": {"name": "Bob", "age": 25}},
  {"type": "edge", "from": 0, "to": 1, "edge_type": "KNOWS", "properties": {"since": 2020}}
]
```

**RDF Import:**
Use the `import_rdf` MCP tool directly. Do not convert RDF to Cypher manually.
```
import_rdf(format: "turtle", data: "<file contents>")
```

**Batch Size Guidance:**
- Small datasets (<100 records): Single UNWIND or individual MERGE statements
- Medium datasets (100-10000 records): Batch into groups of 100-500 records using UNWIND
- Large datasets (10000+ records): Use POST /import API endpoint for best performance

**Idempotency Rules:**
- Always use MERGE (not CREATE) when re-running imports
- MERGE on the smallest unique key set (not all properties)
- Use ON CREATE SET for initial values, ON MATCH SET for updates
- For relationships, MERGE with match on both endpoints + edge type

Target: ~120-150 lines.

**Step 4: Create `skills/data-import/rules/validation-checks.md`**

Data quality validation before import:

1. **Schema Compatibility**: Check if proposed labels and properties conflict with existing schema (via browse_schema)
2. **Uniqueness Check**: Verify that merge key columns have unique values. Warn if duplicates exist.
3. **Type Consistency**: Verify all values in a column are the same type. Flag mixed types.
4. **Null Handling**: Count nulls per column. Warn if >20% of a column is null.
5. **String Length**: Warn if any string value exceeds 10,000 characters (potential data quality issue).
6. **Relationship Validity**: For relationship imports, verify that both source and target nodes exist or will be created.
7. **Encoding Issues**: Check for non-UTF-8 characters, control characters, or malformed Unicode.
8. **Duplicate Rows**: Detect and warn about duplicate records.
9. **Date/Time Formats**: Verify consistent date formats (ISO 8601 preferred).
10. **Numeric Ranges**: Warn about extreme values that might indicate data errors.

**Pre-Import Checklist:**
Before executing import, always present:
- Total records to import
- Schema summary (labels, properties, relationships)
- Validation warnings (if any)
- Estimated number of Cypher statements
- Ask user to confirm before proceeding

**Post-Import Verification:**
After import, always run:
1. `list_datasets` to confirm new node/edge counts
2. `execute_cypher` with `MATCH (n:NewLabel) RETURN count(n)` for each imported label
3. `execute_cypher` with a sample query to verify data is accessible

Target: ~80-100 lines.

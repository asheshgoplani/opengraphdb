<interfaces>
<!-- MCP Tools for Graph Exploration -->
browse_schema: Returns labels, relationship types, property keys (call FIRST)
get_node_neighborhood: N-hop subgraph around a node (for expansion)
search_nodes: Text search across properties (for finding entry points)
execute_cypher: Run arbitrary Cypher (for pattern matching, aggregation)
list_datasets: Database overview with counts (for initial assessment)

<!-- OpenGraphDB Schema Features -->
- Labels: Node classification (e.g., :Person, :Movie, :Airport)
- Relationship types: Edge semantics (e.g., :ACTED_IN, :KNOWS, :FLIES_TO)
- Property keys: Node/edge attributes (e.g., name, age, lat, lon)
- Indexes: CREATE INDEX ON :Label(property) for fast lookup
- Label bitmaps: Internal fast label membership checks (roaring bitmaps)

<!-- RDF Support in OpenGraphDB -->
- import_rdf MCP tool: Imports RDF/Turtle/N-Triples into graph
- export_rdf MCP tool: Exports graph as RDF
- _uri property: Preserved on nodes for round-trip fidelity
- Ontology mapping: RDF classes → labels, RDF properties → edge types or node properties
</interfaces>

Role definition: You are a graph exploration expert for OpenGraphDB. You help users discover, navigate, and understand graph data through systematic exploration.

Content should cover:
- **Your approach**: Always start with schema discovery, then identify entry points, then expand systematically
- **Available tools**: List MCP tools and when to use each for exploration
- **Exploration workflow**:
  1. `list_datasets` to assess database size and contents
  2. `browse_schema` to understand the schema (labels, relationships, properties)
  3. Identify entry points: use `search_nodes` for known entities, or `execute_cypher` with `MATCH (n:Label) RETURN n LIMIT 5` for sampling
  4. `get_node_neighborhood` to expand around interesting nodes
  5. `execute_cypher` for pattern matching and aggregation queries to find structural insights
- **Reporting**: Summarize findings in structured format: schema overview, key entities, relationship patterns, structural insights
- **Rules reference**: @rules/exploration-strategies.md, @rules/schema-navigation.md

Target: ~80-100 lines.

**Step 2: Create `skills/graph-explore/rules/exploration-strategies.md`**

Systematic strategies for exploring unknown graphs:

1. **Top-Down Strategy** (large graphs, 1000+ nodes):
   - Start with schema and label counts: `MATCH (n) RETURN labels(n)[0] AS label, count(n) AS cnt ORDER BY cnt DESC`
   - Identify hub labels (highest count)
   - Sample each label: `MATCH (n:Label) RETURN n LIMIT 5`
   - Map relationship types between labels: `MATCH (a)-[r]->(b) RETURN labels(a)[0], type(r), labels(b)[0], count(r) ORDER BY count(r) DESC LIMIT 20`
   - Deep-dive into interesting patterns

2. **Bottom-Up Strategy** (small graphs, <100 nodes):
   - `MATCH (n) RETURN n LIMIT 50` to see everything
   - Identify clusters via connected components
   - Follow each cluster's relationships

3. **Goal-Directed Strategy** (user has a specific question):
   - Parse the user's goal into entities and relationships
   - Search for entities: `search_nodes` with key terms
   - Expand from found entities using `get_node_neighborhood`
   - Build up the answer through iterative expansion

4. **Pattern Discovery Strategy** (finding structural patterns):
   - Degree distribution: `MATCH (n)-[r]-() RETURN id(n), count(r) AS degree ORDER BY degree DESC LIMIT 10`
   - Triangles: `MATCH (a)--(b)--(c)--(a) RETURN a, b, c LIMIT 10`
   - Star patterns: High-degree nodes
   - Chain patterns: Path queries

5. **Temporal Exploration** (if temporal data exists):
   - Check for timestamp properties
   - Use temporal MATCH: `MATCH (n:Event) AT TIME <timestamp>`
   - Compare versions across time

Include example queries for each strategy. Target: ~120-150 lines.

**Step 3: Create `skills/graph-explore/rules/schema-navigation.md`**

How to use schema information to guide exploration:

1. **Schema interpretation**: How to read the browse_schema response and identify:
   - Central entity types (most connected labels)
   - Junction/bridging types (labels that connect domains)
   - Property richness (which labels have the most properties)

2. **Relationship direction matters**: OpenGraphDB stores directed edges. Always check both directions:
   - `MATCH (a:Person)-[r]->(b) RETURN type(r), labels(b)[0], count(r)` (outgoing)
   - `MATCH (a:Person)<-[r]-(b) RETURN type(r), labels(b)[0], count(r)` (incoming)

3. **Property-based entry points**: When searching for specific entities:
   - Use `search_nodes` for text search across all properties
   - Use `execute_cypher` with WHERE clauses for exact matches
   - Try multiple property keys if the first doesn't match

4. **Building a mental model**: After exploration, summarize:
   - Entity types and their counts
   - Relationship types and their directions
   - Key properties on each entity type
   - High-degree nodes (potential hubs)
   - Isolated nodes or disconnected components

Target: ~80-100 lines.

**Step 2: Create `skills/schema-advisor/rules/modeling-patterns.md`**

Graph modeling patterns and anti-patterns for OpenGraphDB:

**Good Patterns:**
1. **Entity-Relationship**: Core entities as labeled nodes, relationships as typed edges
   - Example: `(:Person)-[:WORKS_AT]->(:Company)-[:LOCATED_IN]->(:City)`
2. **Intermediate Nodes**: For relationships with properties, create an intermediate node
   - Example: `(:Person)-[:HAS_ROLE]->(:Role {title, since})-[:AT]->(:Company)` instead of putting role properties on the edge
3. **Hyperedge Pattern**: When a relationship connects 3+ entities, model as a node
   - Example: `(:Contract)-[:INVOLVES]->(:Party)`, `(:Contract)-[:COVERS]->(:Property)`
4. **Temporal Modeling**: Use timestamp properties for versioning, or OpenGraphDB's temporal query support
5. **Hierarchical Structures**: Parent-child relationships with `:PARENT_OF` or `:CHILD_OF` edges
6. **Tagging Pattern**: `(:Entity)-[:TAGGED]->(:Tag {name})` for flexible categorization

**Anti-Patterns:**
1. **God Node**: One node type with everything as properties. Split into labeled types.
2. **Property Overload**: Storing related data as JSON strings in properties. Create proper graph structure.
3. **Implicit Types**: Using a `type` property instead of labels. Use labels: `:Person`, `:Company`.
4. **Missing Relationships**: Storing foreign keys as properties. Create proper edges.
5. **Redundant Properties**: Storing computed values that can be derived from graph structure.
6. **Over-normalization**: Creating nodes for simple enum-like values. Use properties for small value sets.

**Domain-Specific Templates:**
- Social Network: Person, Post, Comment, Like, Follow, Group
- E-commerce: Customer, Product, Order, Review, Category
- Knowledge Graph: Entity, Concept, Relation (with _uri for RDF)
- IoT/Sensor: Device, Reading, Location, Alert

Target: ~130-160 lines.

**Step 3: Create `skills/schema-advisor/rules/index-strategy.md`**

Index recommendations for OpenGraphDB:

1. **When to create indexes**: Any property used in WHERE clauses for frequent queries
2. **Syntax**: `CREATE INDEX ON :Label(property)`
3. **What to index**:
   - Unique identifiers (name, email, id)
   - Frequently filtered properties (status, type, category)
   - Sort keys (date, timestamp, created_at)
4. **What NOT to index**:
   - Properties with very low cardinality (boolean, status with 2-3 values)
   - Properties rarely used in queries
   - Large text properties (use text_search MCP tool instead)
   - Vector embeddings (use vector_search MCP tool instead)
5. **Composite considerations**: OpenGraphDB indexes on single properties. For multi-property filtering, index the most selective property.
6. **Label bitmaps**: OpenGraphDB uses roaring bitmaps for label membership. Label filtering is always fast. Index properties within labels.
7. **Full-text search**: For substring search, use the text_search MCP tool (tantivy). For exact match, use indexes.
8. **Vector search**: For semantic similarity, use vector_search MCP tool (usearch ANN). Store embeddings as vector properties.

Target: ~60-80 lines.

**Step 4: Create `skills/schema-advisor/rules/rdf-mapping.md`**

RDF ontology mapping guidance for OpenGraphDB:

1. **RDF to Graph Mapping**:
   - RDF classes → Node labels (`:Person`, `:Organization`)
   - RDF object properties → Relationship types (`:KNOWS`, `:WORKS_AT`)
   - RDF datatype properties → Node properties (name, age, email)
   - RDF individuals → Nodes (with `_uri` property preserved)

2. **URI Preservation**: OpenGraphDB stores the original RDF URI as `_uri` property on each node. This enables round-trip fidelity: import RDF, work with it as a graph, export back to RDF.

3. **Ontology Design for Graph**:
   - Keep class hierarchy flat (1-2 levels). Deep hierarchies map poorly to labels.
   - Use OWL object properties for edges, datatype properties for node properties.
   - Domain and range restrictions inform label assignment.

4. **Import Process**: Use the `import_rdf` MCP tool with Turtle, N-Triples, or RDF/XML format. The tool handles:
   - URI → node ID mapping
   - Literal → property value conversion
   - Blank node → anonymous node creation
   - Multi-value properties → multiple property entries

5. **Export Process**: Use `export_rdf` MCP tool. It reconstructs RDF triples from:
   - Node labels → RDF types
   - Node properties → Datatype properties
   - Edges → Object properties
   - `_uri` property → Subject/Object URIs

6. **Common RDF Vocabularies**: When designing schemas for RDF interoperability, use standard prefixes:
   - schema.org (Person, Organization, Event)
   - FOAF (Person, knows, name)
   - Dublin Core (title, creator, date)
   - SKOS (Concept, broader, narrower)

Target: ~70-90 lines.

**traceNodeIds generation:**
For each response, `traceNodeIds` is simply `graphData.nodes.map(n => n.id)` in a logical traversal order. For graph-heavy results, order by connectivity (most edges first) to create an expanding-outward animation effect.

Build a helper function `buildDemoResponse` that takes a dataset key, question ID, cypher string, NL answer string, and a filter function, and produces the complete `DemoResponse`:

```typescript
function buildDemoResponse(
  questionId: string,
  dataset: DatasetKey,
  cypher: string,
  nlAnswer: string,
  filterFn: (data: GraphData) => GraphData,
): DemoResponse {
  const fullData = DATASETS[dataset].data
  const graphData = filterFn(fullData)
  // Order trace nodes: nodes with most edges first for expanding animation
  const edgeCounts = new Map<string | number, number>()
  for (const link of graphData.links) {
    const src = typeof link.source === 'object' ? link.source.id : link.source
    const tgt = typeof link.target === 'object' ? link.target.id : link.target
    edgeCounts.set(src, (edgeCounts.get(src) ?? 0) + 1)
    edgeCounts.set(tgt, (edgeCounts.get(tgt) ?? 0) + 1)
  }
  const traceNodeIds = graphData.nodes
    .map(n => n.id)
    .sort((a, b) => (edgeCounts.get(b) ?? 0) - (edgeCounts.get(a) ?? 0))
  return { questionId, dataset, cypher, nlAnswer, graphData, traceNodeIds }
}
```

Create all ~20 `DemoResponse` entries using this helper. The filter functions should extract relevant subsets:
- Label-based filters: `nodes.filter(n => n.labels.includes('Movie') && n.properties.rating > 4.0)`
- Relationship-based filters: include all edges where both source and target are in the filtered node set
- Cap results at 30-40 nodes for visual clarity

Export `DEMO_RESPONSES` as a `Map<string, DemoResponse>` keyed by `questionId` for O(1) lookup.

**Step 2: Create `frontend/src/lib/demo-trace.ts`**

A utility that generates simulated trace animation from offline query results. Since the demo runs against pre-loaded datasets (no backend), we cannot use the real `POST /query/trace` SSE endpoint. Instead, we simulate the trace animation by feeding node IDs to the graph store's `advanceTrace` at timed intervals.

```typescript
import type { TraceStep } from '@/types/graph'

export interface SimulatedTraceOptions {
  nodeIds: (string | number)[]
  stepDelayMs?: number            // delay between each node lighting up (default: 80ms)
  onStep: (step: TraceStep) => void
  onComplete: () => void
  signal?: AbortSignal            // for cancellation
}

export function runSimulatedTrace(options: SimulatedTraceOptions): void {
  const { nodeIds, stepDelayMs = 80, onStep, onComplete, signal } = options

  let currentIndex = 0

  function tick() {
    if (signal?.aborted) return
    if (currentIndex >= nodeIds.length) {
      onComplete()
      return
    }

    onStep({
      nodeId: nodeIds[currentIndex],
      stepIndex: currentIndex,
    })

    currentIndex++
    setTimeout(tick, stepDelayMs)
  }

  tick()
}
```

This function:
- Takes an ordered array of node IDs (from `DemoResponse.traceNodeIds`)
- Fires `onStep` callbacks at timed intervals to drive the graph store's `advanceTrace`
- Calls `onComplete` when all nodes have been "visited"
- Respects an `AbortSignal` for cancellation (if user switches dataset or sends a new question)
- The 80ms default step delay creates a smooth, visible animation that completes in ~2-3 seconds for 30 nodes

Also export a helper to compute total animation duration:
```typescript
export function estimateTraceDuration(nodeCount: number, stepDelayMs = 80): number {
  return nodeCount * stepDelayMs
}
```

**Implementation details:**

1. **Store access**: Read/write from `useDemoStore` (NOT `useAIChatStore`).

2. **Provider management**: Same pattern as `useAIChat` — hold a `ChatProvider` ref, recreate on settings change, lazy WebLLM init. Import settings from `useSettingsStore`. 3. **sendQuestion(text: string)** — the main entry point:

   a. **Pre-computed fast path**: Check if `text` matches any `DemoQuestion.text` for the active dataset (case-insensitive trim comparison). If match found:
      - Retrieve the `DemoResponse` from `getDemoResponse(question.id)`.
      - Add user message to demo store.
      - Start assistant message in demo store.
      - Simulate streaming with typewriter effect: iterate through the response text (NL answer + Cypher block) character by character with 15ms intervals, calling `appendToMessage` in chunks of 3-5 characters. This creates a convincing "AI generating" feel while being instant.
      - On finalize: set `cypher` on the message, set `graphData` on the store, trigger simulated trace via `runSimulatedTrace`.
      - Total perceived latency: ~1-2 seconds (typewriter) + ~2-3 seconds (trace animation) = well under 5 seconds.

   b. **Live AI fallback**: If no pre-computed match:
      - Add user message to demo store.
      - Set `isLoading(true)`.
      - Ensure provider is initialized (with WebLLM download progress if needed).
      - Build messages array: `buildDemoSystemPrompt(activeDataset)` as system message + conversation history (rolling window of 4 exchanges).
      - Stream via `provider.streamChat(...)`, piping tokens to `appendToMessage`.
      - On finalize: extract Cypher blocks via `extractCypherBlocks`.
      - If Cypher found: attempt to match against offline dataset queries. Use a simple heuristic: parse the MATCH pattern, find nodes/labels mentioned, run the closest existing `filterFn` from the dataset's guided queries, or fall back to showing the full dataset graph.
      - Trigger simulated trace on the matched result nodes.
      - Set `isLoading(false)`.

4. **setActiveDataset(key: DatasetKey)**: Delegates to `useDemoStore.setActiveDataset(key)` which clears conversation and graph data.

5. **Pre-computed text format**: The simulated "AI response" for pre-computed answers should be formatted exactly like a real AI response:
   ```
   {nlAnswer}

   ```cypher
   {cypher}
   ```
   ```
   This ensures the Cypher block renders identically to live AI responses.

6. **Abort handling**: Store an `AbortController` ref. Abort previous request when a new question is sent. Also abort the simulated trace animation when switching datasets.

7. **Return value**:
   ```typescript
   return {
     sendQuestion,                  // (text: string) => Promise<void>
     isReady: boolean,              // true when provider could handle custom questions
     activeDataset: DatasetKey,     // current dataset
     setActiveDataset,              // (key: DatasetKey) => void
   }
   ```

**IMPORTANT implementation notes:**
- The typewriter effect for pre-computed responses must use `requestAnimationFrame` or `setTimeout` (NOT synchronous loops) to allow React to re-render between chunks.
- The simulated trace must call `useGraphStore.setTrace` and `useGraphStore.advanceTrace` (the same global graph store used by GraphCanvas) so the embedded demo canvas animates.
- The live AI fallback for custom questions does NOT execute against a real backend. It generates Cypher and shows a best-effort offline graph result. The experience is: "Here's the Cypher I'd run" + a representative graph visualization.
